## Checker Function — `complete_check` (app/checker.py)

### Responsibility

`complete_check` is responsible for performing one HTTP check and returning structured result data. It does not:

- schedule itself
- query the database
- create or commit a `CheckResult`
- update Redis

Those responsibilities belong to the scheduler or a higher-level orchestration function (`perform_check`, tomorrow's keystone).

### Why separate from persistence

Keeps the HTTP-check logic testable in isolation. The HTTP request can be mocked without needing a database session at all.

### Steps

1. record the start time
2. make the HTTP request
3. capture the response status code, or the request exception
4. calculate latency
5. return data usable to construct a `CheckResult`

### Inputs

- `url` — the endpoint to request
- `target_id` — identifies which monitored target produced the result
- `timeout` — passed in rather than hardcoded, since each target may have a different timeout (`EndpointTarget.timeout_seconds`)

### Measuring latency

Uses `perf_counter()`, a monotonic timer, rather than wall-clock time. Monotonic means it never jumps backward (e.g. from an NTP sync), so it correctly measures elapsed duration regardless of system clock changes mid-request.

Measured around the whole try/except, not inside either branch, since latency is meaningful in both the success and failure case.

`perf_counter()` returns seconds as a float. `CheckResult.latency_ms` stores an integer number of milliseconds, so the conversion (`* 1000`) has to happen _before_ rounding, not after, rounding the fractional-second value first would throw away the precision that matters.

### Successful HTTP response

- `status_code` = the response's status code
- `error_class` = `None`

Receiving a response does not mean the check "passed." A target expecting `200` that returns `500` still completed a real HTTP request, there's a status code and no request-level exception. Comparing against `EndpointTarget.expected_status` is a separate, later concern, not the checker's job.

### Request failure

`httpx.RequestError` is the shared base class covering timeout, DNS resolution failure, connection refused, and network interruption. Caught as one bucket for now, since today's scope is only "response received vs not," not classifying _why_ it failed.

- `error_class` = `type(e).__name__`
- `status_code` = `None`, since no HTTP response was ever received

This required making `CheckResult.status_code` nullable (it was previously `NOT NULL`), matching the pattern already used for `error_class`. New Alembic migration to alter the column; reviewed the autogenerated migration before applying it.

### Returned data

Dict keys deliberately match `CheckResult` columns 1:1, excluding `id` (autogenerated) and `checked_at` (has a server-side default, `func.now()`), so the caller can hand the dict straight to `record_check_result` without reshaping it:

- `target_id`
- `status_code`
- `latency_ms`
- `error_class`

### Persistence flow (target shape, not yet wired)

```text
APScheduler triggers job
→ job loads target configuration
→ job calls complete_check
→ complete_check returns structured result
→ caller calls record_check_result
→ record_check_result commits to PostgreSQL (and best-effort writes to Redis)
```

Committing promptly matters because any in-memory state is lost if the app process crashes.

### Testing strategy

**Tool:** `unittest.mock.patch`, not `httpx.MockTransport` (that's for code using an `httpx.Client` instance; `complete_check` calls the bare `httpx.get(...)` module function directly).

**Patch target:** `"app.checker.httpx.get"`, not `"httpx.get"`. Patch has to target the name as looked up from _inside_ `checker.py`, not where `httpx.get` is originally defined, patching the wrong path silently does nothing.

**`mock.return_value`** — controls what a mocked call returns. Used for the success-path test with `Mock(status_code=200)`.

**`mock.side_effect`** — controls what a mocked call raises instead of returns. Used for the failure-path test with `httpx.RequestError("message")`. Must be an already-constructed instance, not the bare class, if given the bare class the mock tries to construct one itself with zero arguments, which fails since `RequestError.__init__` requires a `message` (this is the actual `TypeError` we hit before fixing it).

**Latency assertion:** `>= 0`, not `> 0` and not an exact value. A mocked call still takes some (tiny) real time via `perf_counter()`, and rounding can legitimately produce `0` for a very fast mocked call.

**Cases actually implemented (two tests):**

- request returns a mocked 200, full dict asserted
- request raises `httpx.RequestError`, `status_code is None` and `error_class == "RequestError"` asserted

**Cases not yet covered, worth adding later:**

- request returns an unexpected HTTP status (e.g. 500) — still a "success path" for the checker, but worth a test proving `error_class` stays `None` even on non-200
- latency asserted as non-negative in both existing tests

Persistence itself (creating and committing a real `CheckResult`) is tested separately, not the checker's concern. Confirmed manually end-to-end: `complete_check` → `record_check_result` → row verified in Postgres via `psql`, correct `status_code`, `latency_ms`, and server-generated `checked_at`.
