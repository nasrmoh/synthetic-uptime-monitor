
# Why Redis and Why Here?

Redis keeps its working dataset in memory, as reading from and writing to memory is fast (compare reading from and writing to the disk).  Though speed isn't our intention, what we want is low-latency access to a temporary operational state. Redis is traditionally described as single-threaded for command execution.

## The separation that defines the Architecture

Postgres is durable, historical, and the source of truth. It can rebuild whatever was lost when using Redis. 

Redis is fast, temporary, and replaceable. 

So why have the cache at all if postgres already has the data? Querying Postgres for the same frequently changing value on every request adds latency and load that a much faster key-value lookup avoids. The reasons that hold up are, avoiding repeated DB queries, reducing read latency, handling values that change a lot, keeping state that is going to change a lot inside of the cache rather than loading it into postgres. It being "faster" is true but not the complete story.


## What does `/ready` Actually Store Check Redis

`/ready` checks if Redis is reachable but `/ready` doesn't actually store readiness. Redis stores operational state that other parts of the app read some separate. 

## Does it Belong on Redis or Postgres?


To determine whether data belongs on Redis, Postgres, or both, its not simply a matter of "speed" but rather the "durability" of data. We can determine this by answering the following

> "If the data disappeared right now, would that be acceptable?"

For our Check Results table this would not be acceptable as it holds our monitoring history.
- Not acceptable, we place this data on Postgres

For the last-known status of a given endpoint target it would be fine for it to disappear. We would just indicate that the data is missing until a new check can be made. 
- Acceptable, We can place it on Redis


So for last-known status we write to Postgres, and then also write the same data (or maybe a truncated version of it) to Redis. 


## Some Alternatives

- *Postgres only* 
	- Simpler, a bit slower but a bit less complex. Works well but adds latency and load we can get reduce using Redis
- *Memcached*
	- A simpler cache than Redis. Has no data structures beyond `key/value`, but no persistence options. A bit less flexible than Redis
- *RabbitMQ / Celery* 
	- real task queues with delivery guarantees and retry semantics. Solving a different problem (async work dispatch), heavier than anything this project needs today. This is also why we're not using a Redis list as a job queue right now: a queue needs guarantees Redis doesn't give for free, and reaching for one quietly reintroduces the durability problem we're trying to avoid.


## In Brief

Redis is an in-memory key-value store used for temporary operational state. For this project, Postgres is the source of truth while Redis stores fast disposable data. For example, the latest status of a monitored endpoint. Even if Redis was to degrade or disappear, the monitoring history would still be preserved in Postgres and from it we could rebuild our cache. This provides a bit of robustness as the application degrades a bit more gracefully. `/ready` reports that Redis is unavailable but our permanent monitoring data remains.




# Redis Learning


## Async Redis

Modern `redis-py` supports both synchronous and asynchronous APIs, as FastAPI is asynchronous, using the async Redis client avoids blocking the event loop


## Basic Redis Operations

`PING`
- verify Redis is reachable
`SET`
- store a value
`GET`
- retrieve a value
`EXPIRE`
- Automatically remove a key after some fixed time


## TTL (Time To Live)
Redis keys can expire automatically

Example

```
SET key value EX 60 
```

After 60 seconds:

```
GET key
```

will return nothing because the key has been removed

TTL is useful for data that will naturally become stale.

## Cache Misses

If a key doesn't exist:

```
GET
```
returns `None`

A missing key is not considered an error because it could mean
- The key expired
- Redis restarted
- The value hasn't been cached yet

The application should try to regenerate the data rather than failing

## JSON serialization

Redis stores bytes / strings

Complex Python objects generally need to be serialized before storing.

For example when writing:
```
Python Object -> JSON -> Redis
```

Or for when reading:
```
Redis -> JSON -> Python Object
```

Simple strings don't require serialization


## Background Tasks

We can update the Redis cache using `FastAPI's` `BackgroundTasks`.

This lets Redis writes happen after the HTTP response has been already returned

God for doing cache updates and light post-response work


## Failed Ping Exception

`redis-py` raises a `redis.exceptions.ConnectionError` on unreachable host, which we will setup to be caught the same way we are catching `SQLAlchemy`'s `OperationalError` for unreachable host to Postgres. 



## Implemented Code for `/ready` 


``` python
@app.get("/ready")  
def ready(response: Response,db : Session = Depends(get_db), rd : Redis = Depends(get_rd)):  
    response.status_code = status.HTTP_200_OK  
    app_status = "ok"  
    rd_status = "ready"  
    db_status = "ready"  
  
    # Try to reach Redis  
    try:  
        rd.ping()  
    except (redis.exceptions.ConnectionError, redis.exceptions.TimeoutError):  
        rd_status = "down"  
        response.status_code = status.HTTP_503_SERVICE_UNAVAILABLE  
  
    # Try to reach Postgres  
    db_test_text = text("SELECT 1")  
    try:  
        db.execute(db_test_text)  
    except OperationalError:  
        db_status = "down"  
        response.status_code = status.HTTP_503_SERVICE_UNAVAILABLE  
  
    if rd_status == "down" or db_status == "down":  
        app_status = "unavailable"  
  
    return {"status": app_status, "dependencies": {"db": db_status, "redis": rd_status}}
```

`rd : Redis = Depends(get_red)` uses a similar dependency injection pattern as the database dependency. 

`cache.py`
```python
import redis
import os
from dotenv import load_dotenv
load_dotenv()

r = redis.Redis.from_url(os.environ["REDIS_URL"], decode_responses=True)
def get_rd():
    return r
```
 But there is an important lifecycle difference between `get_db` and `get_rd`. 

`get_db` creates a new SQLAlchemy Session for each request. Each session contains request-specific ORM and transaction state, which means we don't really want to be sharing it globally between unrelated requests. After each request is completed the session is closed.

`get_rd` actually returns the same reusable Redis client object. Redis client manages the access to a pool of reusable network connections, which means the application does not need to construct a new client for every request.
